Circadian Smart Lighting Platform

SunSync brings a controllable sunrise and sunset indoors, even when the real sun is not available.

SunSync is a smart lighting concept built around an ESP32 controller, ambient light sensing, motion detection, timed light transitions, and a local web interface. The purpose is simple: give the user stronger sleep and wake cues in blackout rooms, irregular routines, or any environment where natural sunlight is not enough.

Instead of acting like a normal bedside lamp, it uses warm red lighting to support winding down and cooler white lighting to support waking up. The system also keeps manual override available whenever the user needs immediate light.

Technical Design

Download original Markdown

The hardware, firmware, and lighting behavior are designed to work as one complete system.

The build centers on an ESP32 with an RGB LED strip, a PIR motion sensor, a potentiometer, toggle switches, three photoresistors, and supporting passive components mounted on perfboard inside a 3D printed shell. The LED strip should use a separate power rail from the ESP32 so the controller remains stable under lighting load.

The firmware is intended for C++ on the Arduino ESP32 toolchain. WiFiManager handles first-time setup, NTP provides time sync, and the embedded web server serves both the local interface and API endpoints such as /api/sunsync, /api/color, and /api/config. During sleep periods, the lamp favors dim red light, and during wake-up it shifts toward brighter cool white light.

  • Three photoresistors help maintain usable ambient readings even if one sensor is blocked.
  • The PIR sensor uses SPI, while the photoresistors connect through analogue inputs.
  • Heat-set inserts make the prototype shell easier to disassemble, revise, and reassemble.
  • The biological goal is to support melatonin release at night and alertness in the morning.

Risk Assessment

Download Word document

The project risks are clear, prioritized, and already tied to practical mitigation decisions.

The highest-risk issue identified so far is power instability from LED current draw, which is why the design separates the LED rail from the ESP32 supply and adds smoothing components. Thermal management is also important, so venting, heatsinking, and brightness limits are part of the mitigation approach for LED heat build-up.

Sensor reliability and project scope remain the main open concerns. PIR false readings can affect sleep detection, and native mobile apps can expand the workload too quickly if they are treated as part of the MVP too early. The current approach keeps the first milestone focused on the lamp and its local web UI.

  • Closed: power instability, WiFi setup friction, and component availability.
  • Mitigated: LED heating and storage pressure on the ESP32.
  • Open: motion sensing reliability and scope creep from optional mobile apps.

Verification

Download Excel workbook

The test plan follows the full journey from first boot through API control, timed behavior, and saved settings.

The current workbook defines fifteen functional test cases. It begins with startup, access point creation, WiFi reconnection, captive-portal fallback, mDNS discovery, and NTP sync. From there, it verifies the local API by checking device identity, current color state, color updates, and configuration updates for both sleep and wake cycles.

The later tests focus on the behavior that matters most to the user: whether the lamp changes color at the correct time during sleep, whether wake-up light ramps correctly, whether it stays stable outside scheduled windows, and whether the saved UUID and configuration persist across reboots.

  1. TC001-TC005: startup, networking, discovery, and time synchronization.
  2. TC006-TC011: API identity, color handling, and configuration updates.
  3. TC012-TC014: time-based sleep and wake transitions, plus out-of-cycle stability.
  4. TC015: persistence of configuration and device identity after reboot.
